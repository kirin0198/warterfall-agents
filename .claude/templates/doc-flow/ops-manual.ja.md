## 2. 起動・停止手順

> _システムの計画的な起動と停止のステップバイステップ手順です。_

### 2.1 起動チェックリスト

1. インフラ前提条件を確認 (データベース、外部サービス、シークレット)
2. 最新コンテナイメージをプルまたは新規リリースをデプロイ
3. データベースマイグレーションを実行 (該当する場合)
4. 依存関係の順序でサービスを起動
5. 全コンポーネントのヘルスチェックが通過することを確認

```bash
# 起動コマンド例 (インフラ成果物の実際のコマンドに置き換えてください)
# docker-compose up -d
# kubectl apply -f infra/k8s/
```

### 2.2 停止手順

1. メンテナンス時間をユーザに通知
2. 新規トラフィックを停止 (ロードバランサールールを更新)
3. 処理中のリクエストが完了するまで待機
4. 逆依存順序でサービスを停止
5. 必要に応じてデータバックアップを実施
6. 停止完了を記録

## 5. インシデント対応

### 5.1 重要度分類

| 重要度 | 定義 | 対応目標時間 | 例 |
|------|------|------------|---|
| P1 (致命的) | システム全体停止 | 15 分 | DB 停止、決済不可 |
| P2 (高) | 主要機能の劣化 | 1 時間 | ログイン障害、データ破損リスク |
| P3 (中) | 軽微な機能劣化 | 4 時間 | 非重要 API エラー |
| P4 (低) | 外見上の問題 / 軽微 | 次スプリント | UI の軽微な不具合 |

### 5.2 インシデント対応フロー

1. **検知** — アラート発火またはユーザ報告
2. **受付** — オンコールエンジニアが SLA 以内で受付
3. **トリアージ** — 重要度評価と影響範囲の特定
4. **封じ込め** — 即時の緩和策を適用
5. **解決** — 根本原因の修正
6. **ポストモーテム** — 根本原因分析と再発防止計画
